기술
Java, Spring Boot, Redis, REST API
결과
SSAFY 특화 프로젝트 우수팀

문제 상황

  • 외부 배포 준비 중 보상 API 악용 가능성을 확인
  • 클라이언트 요청만으로 보상을 지급하면 중복 보상과 비정상 요청 가능
  • 실제 서비스라면 공정성과 신뢰성에 영향을 줄 수 있다고 판단

  • SecureRandom 기반 세션 토큰 생성
  • Redis NX, TTL 옵션을 활용한 검증 로직 설계
  • 서버 측에서 세션을 원자적으로 검증하도록 구성

결과

  • 중복 보상과 비정상 요청을 막을 수 있는 구조로 보완
  • 맡은 기능 구현 이후 운영 중 생길 수 있는 문제까지 확인
  • 프로젝트 우수팀 선정

배운 점

  • 기능이 동작하는 것과 운영 가능한 구조는 다름
  • 백엔드는 요청 처리뿐 아니라 서비스 신뢰성도 지켜야 함
  • 맡은 기능 이후의 문제까지 보는 태도가 중요함
3. Project

JellyPaw

게시글 목록 조회 최적화

N+1, count 쿼리, offset 문제를 나눠 보고 조회 구조를 단순화한 경험
프로젝트
반려동물 건강 종합 플랫폼
형태
6인 팀 프로젝트
기간
6~7주
담당
백엔드 / board / board-view / 게시글 목록 조회 API
기술
Java 17, Spring Boot, JPA, Hibernate, MySQL 8.0, Docker
키워드
Slice, cursor, fetch join, IN 조회, CQRS 경험

문제 인식

  • 게시글은 쓰기보다 조회 비율이 높을 것이라고 가정
  • 조회가 비효율적이라는 점을 확인
  • N+1, 불필요한 count 쿼리, offset 기반 페이지네이션 한계를 분리해서 분석

개선 과정

  • Page 대신 Slice 적용
  • offset 대신 cursor 방식 적용
  • ToOne 관계는 fetch join으로 함께 조회
  • ToMany 이미지 데이터는 게시글 ID 기준 IN 조회로 분리

검증 / 결과

  • EXPLAIN ANALYZE로 실행계획을 확인하며 개선 방향 검증
  • 조회 구조를 더 단순하고 효율적으로 변경
  • 기술을 많이 쓰기보다 병목을 먼저 봐야 한다는 점을 체감

솔직한 맥락

  • CQRS 분리는 실제 운영 데이터로 검증해 결정한 것은 아님
  • 구조를 직접 경험해보고 싶다는 목적도 있었음
  • Kafka 기반 정합성 구조는 팀원이 설계한 환경을 사용했으며, 실패 재처리까지 깊게 이해한 상태는 아니었음
  • 앞으로 더 공부해야 할 영역으로 인식
