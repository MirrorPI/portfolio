2. Project

Re:Run

보상 API 부정 요청 방지

보상 API를 단순 호출 기반으로 처리하면 중복 지급과 비정상 요청 위험이 있어, Redis 기반 일회성 세션 검증 구조를 설계하고 구현했습니다.
형태
SSAFY 6주 팀 프로젝트
담당
백엔드 1인 담당
핵심
Redis 일회성 세션 검증

Problem

  • 게임 종료 보상 API 반복 호출 가능
  • 클라이언트 요청만 신뢰할 경우 비정상 보상 지급 가능
  • 중복 지급으로 서비스 공정성과 운영 신뢰도 저하 위험

Flow

  1. 1Game Start
  2. 2Redis SET NX TTL
  3. 3Game End Request
  4. 4Redis Session Validate
  5. 5Save Result + Reward
  6. 6Delete Redis Session

Architecture Flow

Solution

  • 게임 시작 시 서버에서 sessionName과 sessionSecret 생성
  • Redis에 SET NX + TTL로 일회성 세션 저장
  • 보상 지급 전 Redis 저장값과 요청값 일치 여부 검증
  • 처리 후 세션 삭제로 재사용 요청 차단

Result / Limitation

  • 중복 보상 요청과 세션 재사용 차단
  • 만료 세션 자동 정리로 운영 부담 완화
  • 기능 구현을 넘어 운영 신뢰성까지 고려

클라이언트 기반 검증만으로는 완전한 보안이 아니므로 서버 권위 검증, 이상 탐지, 트랜잭션 처리 추가 필요

핵심: 보상 지급 전 Redis 일회성 세션 검증, 처리 후 세션 삭제로 재사용 요청 차단

문제 상황

  • 대량 데이터 조건에서 게시글 목록 조회 API 지연 사례 확인
  • 기본 목록 조회가 Page를 반환해 전체 개수 계산용 count 쿼리 발생
  • DTO 매핑 과정에서 댓글 수, 좋아요 수, 조회 수, 이미지 조회 쿼리가 게시글마다 추가 발생
게시글 20개 조회 시 최대 82개의 SELECT 발생 1(count) + 1(list) + 20(comment) + 20(like) + 20(view) + 20(image) = 82

문제를 나눈 기준

1Page 반환 문제

전체 개수 계산을 위한 count 쿼리 발생

2offset 문제

새 게시글 추가 시 중복·누락 가능성

3연관관계 조회 문제

작성자·댓글 수·좋아요 수·조회 수 접근 과정에서 N+1 가능성

4Image 컬렉션 조회 문제

OneToMany 컬렉션 fetch join + pagination 조합은 비효율 가능성

핵심: 응답 지연 현상을 페이지네이션 / count 쿼리 / 연관관계 / 이미지 컬렉션 조회 문제로 분해

해결 과정

1Page -> Slice

전체 개수보다 다음 데이터 존재 여부만 확인

2offset -> cursor

createdAt DESC, id DESC 기준으로 중복·누락 가능성 완화

3ToOne fetch join

작성자 등 ToOne 연관관계를 목록 쿼리에서 함께 조회

4Image IN 벌크 조회

컬렉션 fetch join 대신 게시글 ID 기준으로 분리 조회

where board.created_at < :cursorCreatedAt
   or (board.created_at = :cursorCreatedAt and board.id < :cursorId)
order by board.created_at desc, board.id desc
limit :pageSize

결과

Before 82 SELECT

1(count) + 1(list) + 게시글별 연관 데이터 조회

->
After 2 SELECT

board + ToOne fetch join, image IN bulk query

검증

  • board_view 100만 건, image_view 12만 건 테스트 데이터 구성
  • board_view(created_at, id) 복합 인덱스 적용
  • EXPLAIN으로 인덱스 사용 확인
  • Postman 재측정 결과 warm cache 기준 62ms 확인

해석 기준: 27.46초와 62ms는 동일 운영 환경의 A/B 비교가 아니므로, 절대 수치보다 count 쿼리 제거, N+1 완화, cursor 조건 기반 인덱스 적용, 쿼리 구조 단순화에 초점
