2. Project

Re:Run

보상 API 부정 요청 방지

보상 API를 단순 호출 기반으로 처리하면 중복 지급과 비정상 요청 위험이 있어, Redis 기반 일회성 세션 검증 구조를 설계했다.
형태
SSAFY 6주 팀 프로젝트
담당
백엔드 1인 담당
핵심
Redis 일회성 세션 검증

Problem

  • 게임 종료 보상 API가 반복 호출될 수 있음
  • 클라이언트 요청만 신뢰하면 비정상 보상 지급 가능
  • 서비스 공정성과 신뢰성 저하 위험

Solution

  • 게임 시작 시 서버가 일회성 세션 생성
  • SecureRandom 기반 sessionSecret 생성
  • Redis에 sessionName + sessionSecret 저장
  • NX 옵션으로 중복 세션 생성 방지
  • TTL로 세션 만료 자동 처리

Flow

  1. 1Game Start
  2. 2Redis SET NX TTL
  3. 3Game End Request
  4. 4Redis Session Validate
  5. 5Save Result + Reward
  6. 6Delete Redis Session

Result / Limitation

  • 중복 보상 요청 방지
  • 세션 재사용 차단
  • 만료 세션 자동 정리
  • 기능 구현을 넘어 운영 신뢰성까지 고려

클라이언트 기반 요청 검증만으로는 완전한 보안이 아니므로 서버 권위 검증, 이상 탐지, 트랜잭션 처리가 추가로 필요함

핵심은 보상 지급 전에 Redis의 일회성 세션을 검증하고, 처리 후 세션을 삭제해 재사용 요청을 막는 것이다.

문제 상황

  • 대량 데이터 조건에서 게시글 목록 조회 API 지연 사례 확인
  • 기본 목록 조회가 offset 기반 Page 방식이라 count 쿼리 발생
  • DTO 매핑 과정에서 댓글 수, 좋아요 수, 조회 수, 이미지 조회가 게시글마다 추가 발생
게시글 20개 조회 시 최대 82개의 SELECT 발생 1(count) + 1(list) + 20(comment) + 20(like) + 20(view) + 20(image) = 82

문제를 나눈 기준

1Page 문제

무한 스크롤에 필요 없는 count 쿼리 발생

2offset 문제

새 게시글 추가 시 중복·누락 가능성

3연관관계 조회 문제

작성자·카운트·이미지 접근 과정에서 N+1 가능성

4Image 컬렉션 문제

OneToMany + pagination 조합은 비효율 가능성

핵심은 응답 시간이 느리다는 현상을, 페이지네이션 / count 쿼리 / 연관관계 / 이미지 컬렉션 조회 문제로 나눠서 본 것입니다.

해결 과정

전체 개수보다 다음 데이터 존재 여부만 확인

2offset -> cursor

createdAt DESC, id DESC 기준으로 중복·누락 가능성 완화

3ToOne fetch join

작성자, 댓글 수, 좋아요 수, 조회 수를 함께 조회

4Image IN 벌크 조회

fetch join 대신 게시글 ID 기준으로 분리 조회

where board.created_at < :cursorCreatedAt
   or (board.created_at = :cursorCreatedAt and board.id < :cursorId)
order by board.created_at desc, board.id desc
limit :pageSize

결과

검증
